      => No Way to "Fix", as Even in Later GCC Versions it _Still_ shows
         as a Warning, and ADARA is Built with -Werror -Wno-deprecated
      => Rather than go in and try to Fix all the Ancient EPICS References,
         just insert some Gruesome GCC Pragmas around EPICS #include... ;-b
         * #pragma GCC diagnostic ignored "-Wdeprecated-copy"
         * #include <casdef.h>, #include <cadef.h>, #include <fdManager.h>
         * #pragma GCC diagnostic pop

         which we _Can_ (and Should!) fix, so I fixed it. :-D
         * as pointed out so nicely on StackOverflow:
           https://stackoverflow.com/questions/62689847/
           c-g-compiler-error-definition-of-implicitly-declared
         * "You actually don't need a copy constructor nor an overloaded
           assignment operator in your class. Since your class doesn't
           have any allocated pointers or members that can't be serviced
           with a shallow copy, the default copy constructor and default
           assignment operator generated by the compiler will do
           nicely."
         * So I simply _Removed_ the unnecessary Copy Constructors
           from the Header Files and Class Implementation, and It Works!
      => Also Another Deprecated GCC Issue -

   -> Looks like everything "Works" with Log4Cpp in Rhel7 now, but it requires some changes to the age-old SMS "logging.conf" file, to add a "Root Appender" with proper pattern layout formatting, e.g.:

   log4j.rootCategory=DEBUG, rootAppender
   log4j.appender.rootAppender=org.apache.log4j.ConsoleAppender
   log4j.appender.rootAppender.layout=org.apache.log4j.PatternLayout
   log4j.appender.rootAppender.layout.ConversionPattern=%d %p %c - %m%n

   -> Dagnabbit, one more Log4Cpp issue, on Rhel9, where the latest GCC/G++ Compiler 11.3.1 is yet a step more pedantic and is somehow losing track of _Which_ Logger is Which. This only happens for the infamous ReadyAdapter class, which is header-only based. When the SMSControl class tries to create a ReadyAdapter, the class somehow does _Not_ inherit the "correct" stupid static global "logger" reference from SMSControl, but instead some other as-yet-uninitialized "logger" that's NULL (0x0) and therefore causes an unavoidable segmentation violation. ;-Q  The easiest solution for this is to simply split ReadyAdapter.h into a proper *.cc source file, so it can create its _Own_ Log4Cpp stupid static global "logger" instance, and be sure to initialize it before use. ;-D  This worked and was the "least grotesque" alternative to occur to me at the time. Lol... ;-D

   export LESS="-R"

   -> This provided true Color Diff output in a terminal shell. Yay! :-D
